Getting Started with Cargo

After reading this guide you will know:

  • How to install and configure your Cargo application server.
  • The general structure of a Cargo application.
  • How to invoke actions on your server from a javascript client.
  • How to create a simple application that uses basic parts of the framework.

Prerequisites

To start developing web applications with Cargo, you will need some knowledge of HTML, CSS and JavaScript. Because the back-end of Cargo was witten with Go, it and has no external dependencies so there nothing more than Cargo itself to install. However, you will not need to know or use Go to build a web application with Cargo.

Installing Cargo

Unzip the file where you want your sever to be and voîlà! you have a server ready to run.
To start the server simply call the command ./CargoWebServer in linux, or CargoWebServer.exe in windows. Note that the server might take several minutes to load the first time.
When the indexation is done, Bridge will open. To login, in the user name input enter admin and in the password input enter adminadmin.

In order to work properly, the WebApp folder and the sever executable file should be at the same directory level. If you need to separate these tow files, you must define a global variable named CARGOROOT and make it point to WebApp directory. The application directory Apps contain applications files. From the browser point of view it's the root of your server file system. With a running server, clicking the link "Apps" display the Apps folder content.
Here's a list of server directories description:

  • The lib folder is the place to put external libraries used by all your applications. E.g. underscore, jQuery, ...
  • The Cargo folder contain the code used by all applications to interact with the server. We will see later which files must be included by application.
  • The Data folder contain applications data. The MimeTypeIcon directory contain icon's named (mime type extension).png
    ** Never delete files colorName.csv and mimeType.csv
  • The Schemas folder contain XML schemas used by applications.
    ** Never delete files xs.xsd and SqlTypes.xsd
  • The Script folder contain sever side JavaScript files.

Front End

Element

The 'Element' class included in the Cargo javascript framework offers a basic Html element wrapper class to help you create client side graphical interfaces. It gives you the ability to create and manipulate html elements within your javascript files rather than in the traditionnal html and css files. All the functionnalities related to this class can be found within the file 'element.js'.

You do not need to use the 'Element' class or any of it's related functionnalities to develop a web application using the Cargo framework. It can be used in conjunction with other javascript libraries such as Jquery and Bootstrap. If you are an experienced front-end web application developper and already have techniques which you are familiar with, feel free to skip to the next tutorial.

Element essential functions

To create a new element, simply call it's constructor function. 

var myElement = new Element(parent, params, callback, appendFront)
  • parent Parent element of this element. Can be an Element or a DOM element.
  • params List of parameters.
  • callback Function called after the initialization is completed.
  • appendFront 'true': the element is put in front of the other elements. 'false': it will be at the end.

This function is used to append a child element to an existing element. 

myElement.appendElement(childElement)
  • childElement The element to append. It can be an existing element, or a list of element properties.

This function is used to remove a child element from an existing element. 

myElement.removeElement(childElement)
  • childElement The child element to remove.

This function is used to find a child recursively inside the element's childs hierarchy. 

myElement.getChildById("childElementId")
  • id {string} The id of the element to retreive.

This function is used to navigate downwards in the Element hierarchy. It returns the last child of the element. 

myElement.down()

This function is used to navigate upwards in the Element hierarchy. It returns the immediate parent of the element. 

myElement.up()

Asynchronous programming

Asynchronous programming means that your web application can execute multiple segments of code simultaneously. To do so, Cargo uses 'callbacks'. A callback is a function that can be passed as an argument in another function. Cargo has three types of callbacks.

  • Progress callbacks: executed during the progress of the function. (For example used as feedback when transfering a file through the network).
  • Success callbacks: executed when the function has succeded.
  • Error callbacks: executed when the function fails.

Callbacks are in most cases optional parameters, meaning that if you can pass an empty function (function(){}) or undefined as a callback.

Some functions have a parameter named 'caller'. The caller is a JSON in which you can store object from the request context and get it back from the response context. This means you can place a variable in the 'caller', pass it as a parameter to the function, modify it in the callback functions, and retrieve it afterwards.

Example

Session manager's login function

var user = {
	"userName" : "myUserName",
	"password" : "password01"
};

server.sessionManager.login(
	/* name */
	user.userName,
	/* password */
	user.password,
	/* Success Callback */
	function (result, caller){
		console.log(caller.userName + " logged in.");
	},
	/* Error Callback */
	function (err, caller) {
		console.log("Login error ", err);
	},
	/* caller */
	{ "user": user}
);

Note that the 'login' function has a success callback, an error callback and a caller but it does not have a progress callback.

Chosing a framework simply because it's the flavor of the day whiout knowing to much about it is a risky buisness. Extensibility, simplicity, efficiency, flexibility are overused word's these day's. So let's put all of this in perspective and find out if Cargo is the good tool for you.

First of all simplicity. My background is OOP the first programming language I learn was C++, and later Java. In a language like JavaScript structures are more a matter of desing, but trust me without structures, simplicity give place to complexity. Philosophy

In tow words non-intrusive. Cargo dosen't impose you any rules. The role of Cargo is give you access to server functionalies via the server object. Simplicity here is achieve by making abstraction of the network. We also create some helper class to simplify the use of HTML and SVG element. In order to keep thing simple, we keep external dependencies at mimimum. You can use tool like jQuery, Bootstrap or Angular but those libraries are not part of the framework. We will see later how to achieve extensibity whit Cargo.

Architecture

From desktop application to web application

Application programming in Java, C++ or objective-C fowllow the same principles. Framework in these languages offer basicaly the same kind of functionalities like database access, interface design tools, openGL api... They are composed of various libraries that exetend functionalities of the language and facilitate the development of application. Swing in Java, Cacao in Objective-c, MFC and Qt in C++ are good example of desktop application framework. If we think HTML5 as a JavaScript framework we can see some similarities with these frameworks. Each new functionnality added in HTML5 address a need of application programming. With all these new functionalities we can now considere JavaScript/HTML5 as a desktop application framework.

There's no formal definition of what is a web application. For the sake of clarity, let's define web application as distibuted application. The major difference between a desktop application and a web application is the model of execution. Desktop application run on single computer, and distributed application on a network. In those applications, functionalities are access with use of libraries. The communication with the framework and the application are made via object instances or function call. Instead of libraries, distributed application made use of services, and call to function are made via RPC or reference to distributed object.

Services offer by Cargo are, objects persitence, sessions management, events management (channels, listener), ressource management (files, user, group, role, action...), data access (SQL, Key/Value), authentication (LDAP). Because Cargo take care of all the burden of distributed aspects of your application, developping web application with it isn't more difficult than developping desktop application.

The big picture

Cargo simplify web application programming by exposing distant services as local objects. Just like you use document object to intereact with HTML elements, you use server object to interact with the backend. The only difference is the communication model. Local object methods call are synchronous and distant object method call are asynchronous. To work around it and simplify the code, we made use of callback's functions. The three callbacks functions are, success, error and transfer. They represent the state of the action call on the backend. In backround, when a method on server object is invoke, a request to the backend is send. The request is process by the backend and a response message is sent back to client. Depending of the response type, the appropriate callback function is call.

Distributed system based from OOP made use of interface definition language IDL to describe component's application programming interface. With Cargo, in place of IDL, we made use of JavaScript. Because Cargo backend can interpret JavaScript and because all of it's functionalities are interface with JavaScript, you can ask Cargo to do all you want. Following the the open closed principle, you can exetend the server functionalities from the client side without modifying it's code.

Event model, don't call us, we'll call you

Cargo event model are pretty simple and pretty usefull. The desing of the event system follow the publish-subscribe pattern principles. Event can be see as piece of inforamtion travelling through channels. Each channel can carry one type of event and each event can travel throw one channel. Channel are dynamic, when a listener register itself on an event channel, the channel is created, and inversely the channel is close when no more listeners use it. One of the key concept in Cargo event model is the EventHub. From the point of view of server, who play the role of publisher, the EventHub play the role of subscriber. But from the point of view of local object, who play the role of subscriber, it play the role of publisher. Just like event channels and events, EventHub are specialize in on type of event. Class derived from EventHub play the role of event controller in your application. Here we made use of a variation of the observer pattern. If an object want's to be inform of a given event, it define an update function for that particular event and attach itself to the EventHub reponsable of that event. Latter when the object has no more interest to receive event message, he simply detach itself from the EventHub. Because there can be a lot of event travelling the network, we create event filters. Event filters are regex regular expression. By default channel are silent, in order to receive event you must register a regular expression (filter) for a given event type. To get all event of a given type the regex * can be use. By using the event model in your application you decoupling classes and making your code more robust and reusable.

Entity, something that exists as itself

JavaScript Object Notation JSON is the facto information model those days. Shinning by it's simplicity and it's versatility every structures can be serialyse with it. But when you serialyse an instance of a class and instantiate back you end up with a json object and not the initial instance. That represent a problem when it's time to make object persistent. Every data store need schema to represent the informations structures, XML and SQL has it. The goal able to represent informations structures from SQL and XML without lost.
